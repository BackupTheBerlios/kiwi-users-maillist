Hi,

> The thing is that we have PCs, with good enough CPU performances, Hard 
> drive (80GB) and heavy applications to run.
> We are not trying neither willing to run applications on a central server.
> Our goal is only to distribute during each reboot a clean image of an OS 
> on our students computers.
> If a part (or all) of the image can be NFS-ro mounted, then we boot much 
> quicker. And the unionfs/aufs would allow to write the files needed 
> during the use.
> 
> Do you know if LTSP can do that also, or any other solution.

The problem you are facing here is that you try to mount aufs with
a NFS layer in between which is really problematic. I doubt it will
work reliably. I established a similar system at the university I have
studied but it doesn't use NFS root mounts. I have created a compressed
squashfs image of the system and took care for a relatively fast
network bandwidth (100Mbit with fast backbone). According to this
the compressed image could be transfered rather fast on each computer
and you can change the operating system on the machines one hour
before the class starts. The image itself is mounted unified so all
write operations takes part on a seperate partition of the disk.

We tell the students to store their data on a seperate medium because
these systems are re-installed day by day. This serves the purpose
but I don't know if it would work for you too. At the Siemens
training center they have a similar system but they spent a lot more
money on a SAN system to be able to deploy images faster and on more
machines but the design is the same

If you consider that as a solution kiwi is appropriate to create the
images
