Hi,

> my custom gfxboot-branding is now ready and installs it's files into
> /etc/bootsplash/themes/OX.

This is wrong, your custom gfxboot branding has to follow the
guidelines of the _gfxboot-branding_ packages and not the
bootsplash-branding package

gfxboot     -> graphics screen in grub and isolinux
               Follow the rules of the gfxboot-branding packages

bootsplash  -> graphics background of the framebuffer console
               Follow the rules of the bootsplash-branding packages

> I also created an own version of the boot stuff suse-11.1 and changed
> images.sh to call
>
> suseGFXBoot OX isolinux
>
> instead of openSUSE.

That's also evil: Don't create your own copy of the boot image if you
want me to help with problems. You need to use the boot-theme
attribute in your system image config.xml

<preferences>
     ...
     <boot-theme>OX</boot-theme>
</preferences>

> Unfortunately that does try to compile the gfxboot stuff again in
> /usr/share/gfxboot (from gfxboot-devel), where my branding package does not
> put any stuff, because it is in /etc/bootsplash/themes/OX.

If you have branding packages you are not allowed to have gfxboot-devel
installed. The branding packages has to provide all that what would
be the result of the gfxboot-devel compile code. If you install branding
and devel packages kiwi prefers the devel data and thus your branding
packages have no effect

> How is that designed to work? Do I have to create a package which puts that
> stuff also to /usr/share/gfxboot/themes? Or is there some other way?

* use boot-theme
* don't copy any boot image code !
* follow the rules of the existing gfxboot-branding and/or bootsplash-branding
  packages. Best way is to install the src rpm, exchange the pictures,
  spec file etc.. and rebuild
* don't add gfxboot-devel if you have branding packages

Have fun :-)
