Marcus Schäfer wrote:
> Hi,
> 
> I'd like to discuss the support for USB stick images
> and also whether we can drop the "old way" of creating
> USB stick images.
> 
> in the past we had several improvements with for example
> clicfs and als hybrid ISO's which are both an iso and a disk
> in one image. According to this we have three possible
> rocedures to create an image for a USB stick today
> 
> 1) use <type image="usb" .../> to create an usbboot based image.
>    Plug in the stick and call kiwi to deploy it on the stick:
>    'kiwi --bootstick initrd --bootstick-system image'.
> 
> 2) use <type image="oem" .../> to create an oem virtual disk
>    image. Plug in the stick and dump the raw disk on the stick:
>    'dd if=image of=/dev/stickdevice'
> 
> 3) use <type image="iso" hybrid="true" hybridpersistent="true" .../>
>    to create a hybrid iso image. Plug in the stick and dump the
>    iso on the stick: 'dd if=image of=/dev/stickdevice'
> 
> The first image type is imho obsolete. I'd like to know
> if there are people who still use the image="usb" type. Please help
> me to find out if we can safely drop this code. 

My concern is the partitioning option. Yes, it's been 6+ month since I 
had a chance to look at this but I think it is a valid use case.

The use case is that one wants to partition the stick such that the 
first partition is of format VFAT. This allows a limited area of the 
stick to be used to transport data. If the first partition on a stick is 
not VFAT the software abomination from Redmond wants to erase the stick.

AFAIK this partitioning cannot be implemented with option 2 & 3, but can 
be implemented with option 1 (providing I find the time to get it 
implemented).
