On 09/27/2010 03:15 PM, luis.arruda at emc.com wrote:
> Hi everyone,
> 
> I'm working on a project to implement a "flip-flop" upgrade for an appliance based on a Kiwi OEM-image installation. A "flip-flop" upgrade works thusly:
> 
> 1) There are two partitions on the boot drive (not including swap or other common areas). One (to be referred to as the "old") contains the currently booted OS. The other (to be referred to as the "target") is either empty or contains an obsolete operating system installation due to a prior upgrade.

The data separation example
(http://en.opensuse.org/SDB:KIWI_Cookbook_Data_Separation) shows how to
create a separate partition that could be used for this concpety.
> 
> 2) The new operating system is installed to the "target" partition.
> 
> 3) The boot loader configuration is modified so that the "target" partition is now the default boot partition and the "old" partition is still available via the boot menu should the user want to revert.
> 
> This is to be accomplished using a common Kiwi image that can also be used to install to virgin, completely blank hard drives. 

For the bare metal install you definitely want to use an OEM image, ISO
or USB.

> I've tried just manually extracting the raw image data that's created by our Kiwi scripts/configuration but it doesn't contain a proper initrd to boot off of,

The raw image is a complete disk image, you can test it with

# qemu-kvm file-name.raw

You can dump this image to any disk and boot it and it will work.

> which leads me to believe that this part is handled by the Kiwi code that installs

The kiwi install code for an OEM image basically just does a "dd" operation.
