> Background: When deploying openSUSE to a new machine, I'm using
> a handmade script which makes and mounts some filesystems
> on the fresh hard disk and then essentially runs
>  $ zypper --root /mnt/new_root in pattern:enhanced_base ...
> to install a minimal system.  Judging from the logfiles kiwi
> apparently does something very similar, but the script which
> calls zypper seems to be much more well-thought-out (using
> chroot/lxc) than my simple zypper command line above.  I tried
> to replace my zypper line with a kiwi call, with appropriate
> config.xml, but kiwi refuses to fill the mounted soon-to-be
> root file system, it insists on creating a new directory itself.
>
> Also, I'm wondering if the kiwi-approach to deploy SUSE
> installations outlined above brings any pitfalls which I might

To answer your questions:
> If /usr and /boot are mounted to a different file
> system and you let kiwi prepare the image into
> this root tree, what is the expected outcome?
Essentially I expect a bootable and working system.
Apparently that's what I get: The system which I'm currently
working on has been installed with kiwi's prepare-step
after I manually patched kiwi to
 * add the --download-as-needed option to zypper, and
 * permit preparation into non-empty root directories.

> A system you can chroot into?
Currently I chroot into the system after kiwi is done.
There I run things like grub2-mkconfig and mkinitrd and
rewrite /etc/{fstab,crypttab}.  But I am considering to move
these things into kiwi's config.sh somewhere in the future.

> And the /usr and /boot is shared across many systems?
No, they are on a standalone machine.
They need to be on separate filesystems because they
are backed by a fairly complex combination of lvm
(with mirroring/striping) and luks.
