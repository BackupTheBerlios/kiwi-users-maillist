> 
>    1) control how to boot, that's the boot image the initrd
>    2) define what the real system should be able to do, that's the system image

What I forgot to mention: point 1) is the service kiwi provides. Which
means for all image types kiwi provides boot images. For the special use
cases we try to provide profiles but you can imagine there might be
special cases which are not covered. In such a case people most often
start to clone from an existing boot image and add their special changes.
That's ok of course but also means people need to take care for the boot
and the system image.

If everything is just fine the idea is that people only take care on
the system image and according to this they only take care for the most
important part. Nobody should care about how that thing boots that's
kiwi's job ;) Unfortunately sometimes the requirements also steps into
the boot part and so the boot image description needs to be adapted
which should be avoided because it's a very sensitive part of an image
but if urgently needed one can adapt it too.

So the intention is to provide as much as possible and make it easy
but don't loose the flexibility if the default doesn't apply.
