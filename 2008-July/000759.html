The best way is to package your software put the result into a
directory and set that directory along with a coresponding <package>
line as additional repository in your config.xml

That's the best and the cleanest solution but also the most time
consuming one.

If I had to compile from source I also do that within the new root
tree (chroot). This means I have to have all software to compile the
program installed in my new root tree. People might say this is bad
because the devlopment kit is needed only to compile but not to
run the software. That's true but it's a clean way to compile the
code

The solution you've chosen works too but it's easy to stumble over
multiple possible traps:

- if the host you compile the software on is not on the same version
  compared to the system in your new root tree, the compiled binary
  could work but don't have to

- the configure call resolves the buld environment correctly but
  there is no guarantee that this requirements are also correct in
  your new root tree. If you copy the tree after make and run make
  install inside the new root you risk a broken installation.
  missing libraries are only one problem there could be other
  programs/configurations missing.
   
I think if it's your own software or a small program you can go
that way but don't try to do that with for example openoffice :-)
There are also many public repositories out there which package nearly
every program.
